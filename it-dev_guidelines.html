Linee guida per gli sviluppatori

Coinvolgimento

Tutti sono i benvenuti quando offrono un po' di aiuto. Abbiamo bisogno che questi compiti siano eseguiti:

  • testing di rilasci alpha e beta
  • segnalazione, riparazione e test dei bug
  • scrittura della documentazione (sia documentazione del codice che manuale utente)
  • scrittura di traduzioni dell'applicazione
  • revisione di codice e documentazione
Tutti questi compiti sono ugualmente necessari, quindi sentiti libero di contribuire nel modo che ritieni più adatto a te!
Poiché il progetto è ancora piccolo, abbiamo solo alcune mailing list: sviluppo, bugs e annunci. In prima istanza, gli svilupatori sia della documentazione che delle traduzioni si iscriveranno alla mailing list di sviluppo.

Android

Scrittura del codice

Alcuni semplici consigli e regole:

  • Scrivi bene il codice. Altre persone lo leggeranno, quindi rendilo il più possibile leggibile e auto-esplicante. Non usare nomi di variabili di una sola lettera (tranne che per indici di array e cose simili), prediligi nomi significativi camelCase. Di contro, non usare nomi che sembrano infiniti - alcuni sviluppatori potrebbero preferire la modifica tramite blocco note.
  • Usa i commenti. È un piccolo sforzo per chi scrive codice spiegare cosa il codice fa, può essere un grande sforzo per gli altri leggerlo.
  • Usa Doxygen per i commenti ai metodi e alle classi, in modo da poter creare automaticamente la documentazione del codice e permettere agli altri di usare la tua classe - una usata, una da mantenere.
  • Scrivi librerie e utilità in modo che siano general purpose. Prova a immaginare le varie funzionalità che i tuoi metodo possono avere, e scrivi la firma del metodo come se le fornisse tutte - non importa se non le implementi tutte subito.
  • Usa i TODO se non hai tempo di implementare tutte le funzionalità che un metodo può avere. Concentrati sul tuo compito principale, altre persone faranno evolvere il tuo metodo.
  • Preserva gli attributi compilabile ed eseguibile per l'applicazione nel suo complesso. Una volta eseguito il commit, le tue patch non dovranno mai e poi mai far fallire la compilazione o mandare in crash l'applicazione.
  • Ricorda gli effetti collaterali quando modifichi il codice di altri. Non c'è nulla di peggio di creare bug perché la firma di un metodo è cambiata.
  • (Prova a) intercettare sempre le eccezioni. L'applicazione dovrebbe bloccarsi in modo elegante. (Sarebbe meglio che non si bloccasse del tutto, ma gli sviluppatori possono non essere sempre perfetti.)
  • Poni sempre una variabile private static final String LOG_PREFIX = <class>.class.getSimpleName() all'inizio delle tue classi, in modo da permettere agli altri programmatori di definire in modo semplice i propri log se necessario.

Convenzioni di denominazione

  • Le Activity devono essere chiamate <SempliceDescrizioneDiQuelloCheFa>Activity, ad esempio, StartActivity, VotingActivity, ClassificationActivity. I layout defono essere nominati activity_<semplice_descrizione_di_quello_che_fa> or fragment_<semplice_descrizione_di_quello_che_fa>.
  • Le utilità devono essere nominate come POJO.
  • Si applicano le comuni convenzioni di denominazione di Java.

Compiling

As a typical Android application, the most-used compiling will probably happen inside a preconfigured environment, such as Eclipse with Android SDK or Android Studio. In both cases, the IDE provides one-click buttons to both compile and install the application on a plugged Android device. Should such an environment not be available (or, for command-line addicted), it is possible to create and compile Android applications without Eclipse or Android Studio. Please check Android official documentation.

Installing

Again, without a preconfigured environment, it is enough for the user to upload the APK file (the main result of Android application building) on the selected device, and double-click... better, tap the file icon. Auto-install should begin, and at the end of the process you will be able to open and run the application.
Remember to provide permissions to install application coming from "e;unknown sources"e; - i.e., not coming from Play Store. We're not ready for a full publishing yet, and we will only provide stable versions. Testers will have to manually download the APK from download page (when it will be provided).

Running

In the very first release, the application is only capable of summing some marks for some competitors.
Data will be loaded from a default XML file, when no data.xml file is found on SD card. Tap on the competitor name to show her votes - or to give her votes, and tap the text box to enter a value.

Testing

In this very basic scenario, the tester will just have to check the votes are correctly summed, and the classification is correct. These points will be found in every checklist written for testing purposes, just to be sure basic features are always implemented.
